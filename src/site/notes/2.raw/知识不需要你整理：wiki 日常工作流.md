> 你负责筛选和提问，LLM 负责编译和维护。你不需要整理笔记，但知识体系在持续完善。

我的 Obsidian vault 里有一个 wiki 层，由 LLM 自动维护——我负责筛选源文件和提问，它负责消化内容、建立关联、持续更新。整个系统的设计思路在[[2.raw/让知识复利：用 LLM 建一个会生长的个人 wiki\|让知识复利：用 LLM 建一个会生长的个人 wiki]]里讲过了，这篇说明日常工作流。

## 日常工作流

### 收集素材

看到好文章、有价值的网页，用 Obsidian Web Clipper 或手动保存到 `1.inbox/` 下对应的子目录：

| 子目录 | 用途 |
|---|---|
| `1.inbox/draft/` | 自己的原创写作 |
| `1.inbox/clippings/` | 网页裁剪、外部收藏 |
| `1.inbox/work/` | 工作笔记 |
| `1.inbox/life/` | 生活记录 |

这一步不需要 LLM 参与。你只管往里扔，不用整理格式，不用想 " 这篇文章该归到哪个主题 "——那些都是后面的事。

### 审阅素材

当你确认某篇文章质量合格、值得长期保留时，手动将文件从 `1.inbox/` 移到 `2.raw/`。不值得保留的直接删除。

这一步也不需要 LLM。这是你的质量门控——只有你确认过的内容才会进入 wiki 的源文件层。**inbox 里有很多质量参差不齐的内容，如果让 LLM 直接处理 inbox，低质量内容会污染 wiki。** 所以审阅权留在你手上。

### 触发摄入

文件移入 `2.raw/` 后，在 Claude Code 中对 LLM 说：

| 你说什么 | LLM 做什么 |
|---|---|
| `ingest` | 自动找出所有未被 wiki 覆盖的 `2.raw/` 文件，批量处理 |
| `ingest xxx文章` | 只处理指定的那一篇，会先和你讨论要点再写入 |

**交互模式**（推荐新手使用）：指定单篇文章，LLM 会先读完文章，和你讨论 " 这篇文章的要点是什么、值得强调什么 "，确认后才写入 wiki。你对结果有更多控制。

**批量模式**（适合已熟悉流程后）：直接说 `ingest`，LLM 自动处理所有新文件。你可以事后在 Obsidian 中浏览 wiki 页面，检查更新是否符合预期。

一次 ingest 可能同时影响五到十个页面——更新已有主题页的相关章节、创建新的概念页、补充实体页的文章索引。就像给一棵树浇水，水不是简单倒在地上，而是被根系吸收，输送到每一片需要的叶子。

### 浏览结果

ingest 完成后，打开 Obsidian：

- **看 index.md**：哪些页面更新了，哪些是新建的
- **看 log.md**：最近一次 ingest 的详细操作记录
- **点进具体页面**：阅读更新后的内容，检查综合是否准确
- **用 Graph View**：查看页面间的关联关系

如果发现 LLM 的综合有偏差，直接在对话中告诉它修正。

### 日常提问

不需要任何准备，直接用自然语言提问：

- AUTOSAR 的通信栈分层是怎样的？
- 王垠对 TypeScript 的主要批评是什么？
- 函数式编程和面向对象编程，各自的优劣势？

LLM 会先读 index.md 定位相关页面，然后读取具体的 wiki 页面。如果 wiki 页面不够详细，再去读 `2.raw/` 中的源文件。回答时附上 wikilink 引用，方便在 Obsidian 中跳转。

**如果回答本身有价值**（比如一个跨主题的对比分析），LLM 会建议存为新的 wiki 页面。这类跨主题综合正是 wiki 最有价值的部分——认真考虑这个建议。

### 定期巡检

隔一段时间（比如积累了 20-30 篇新源文件后），对 LLM 说 `lint` 或 `巡检`。LLM 会做全量健康检查：页面间是否有矛盾、是否有孤立页面、是否有知识缺口、源文件计数是否准确。每次 ingest 后 LLM 也会自动做局部巡检，但全量巡检需要你主动触发。

## 什么时候做什么

| 时机 | 你做什么 | LLM 做什么 |
|---|---|---|
| 看到好文章 | Web Clipper → `1.inbox/` | — |
| 文章审阅完毕 | 手动移到 `2.raw/` | — |
| 有新文件入库 | 说 `ingest` 或 `ingest xxx` | 读取、综合、更新 wiki 页面 |
| 想了解某话题 | 直接提问 | 查 wiki、综合回答 |
| 积累了一批新内容 | 说 `lint` | 全量健康检查 |
| 发现 wiki 有错 | 直接告诉 LLM 修正 | 更新对应页面 |

## 几个要点

### 不用记命令

LLM 能理解各种说法：

- `ingest`、`摄入`、`处理新文件`、`帮我把新文章整理进去` → 都会触发 ingest
- `lint`、`巡检`、`检查一下 wiki`、`看看有没有问题` → 都会触发 lint
- 任何问题 → 自动走 query 流程

### 用 Graph View 发现关联

每次 ingest 后打开 Graph View 看看：新页面和哪些已有页面连在一起了，有没有孤立节点，哪些页面是 hub。这是 wiki 作为一个知识图谱最直观的呈现方式。

### 不用担心犯错

- `2.raw/` 是不可变的，LLM 永远不会修改源文件
- `3.wiki/` 的所有改动都可以在 Obsidian 中用 git 回退
- 你随时可以让 LLM 修正任何 wiki 页面的内容

**源文件是安全的，wiki 页面可以随时回退，你只需要做一件事：决定什么值得读。**
