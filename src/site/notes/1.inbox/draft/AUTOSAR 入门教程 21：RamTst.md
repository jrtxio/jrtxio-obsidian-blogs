> 理解 RAM 测试的主流算法族、前台与后台两种执行模式的权衡，以及如何根据 ASIL 等级选择合适的覆盖率方案。

RAM 是 ECU 运行时存储变量的地方——如果某个 RAM 单元发生了 Stuck-at Fault（固定为 0 或 1），程序读到的就是错误数据，可能导致控制逻辑失控。在 [[3.wiki/AUTOSAR\|AUTOSAR]] 功能安全框架下，RamTst 模块专门负责检测这类存储器硬件故障。它不是测试 RAM 中存储的内容是否正确，而是测试物理 RAM 单元本身能否正确地写入和读出。

## 测试算法与覆盖率

RamTst 支持多种测试算法，核心思路一致：向 RAM 写入已知的测试模式（pattern），读回后与期望值比较。不同算法的差异在于测试模式的复杂度和覆盖的故障类型。

### March 算法族

March 算法是 RAM 测试中最主流的方法族。它的基本操作是按地址递增或递减顺序，对每个存储单元依次执行写入和读取操作。区别在于"走几遍"和"每遍做什么"。

**March C-** 覆盖率约 90%，执行时间适中。算法步骤为 `↗(w0); ↗(r0,w1); ↗(r1,w0); ↘(r0,w1); ↘(r1,w0); ↗(r0)`，其中 `↗` 表示地址递增，`↘` 表示地址递减，`w0`/`w1` 表示写 0/写 1，`r0`/`r1` 表示读期望值为 0/1。6 遍扫描，能检测 Stuck-at Fault、Address Fault、Transition Fault 和 Coupling Fault。

**March B** 覆盖率约 90%，某些遍中加入了连续的读写操作，对 Coupling Fault 的检测更敏感。

**GALPAT（Galloping Pattern）** 覆盖率接近 99%，但执行时间极长，复杂度为 O(n²)。它对每个单元都执行"写反、遍历所有其他单元读检查"的操作。适合小面积高可靠性区域的测试，不适合大容量 RAM。

### Checkerboard 与 Transparent

**Checkerboard（棋盘格）** 将 RAM 单元按棋盘格模式交替写入 0 和 1，覆盖率约 60-70%，执行速度快。它主要检测 Stuck-at Fault 和相邻单元间的短路故障，但无法有效检测 Coupling Fault。

**Transparent 算法** 是非破坏性测试的代表。先用 March 算法的变体读取当前 RAM 内容并计算签名（signature），测试完成后再比对签名是否一致。由于不修改 RAM 原始内容，适合在运行时测试存放关键变量的区域。

| 算法 | 覆盖率 | 复杂度 | 破坏性 | 适用场景 |
|------|--------|--------|--------|----------|
| Checkerboard | 60-70% | O(n) | 是 | 启动时快速扫描 |
| March C- | ~90% | O(n) | 是 | 前台/后台常规测试 |
| March B | ~90% | O(n) | 是 | Coupling Fault 敏感场景 |
| Transparent March | ~90% | O(n) | 否 | 运行时变量区测试 |
| GALPAT | ~99% | O(n²) | 是 | 小面积高可靠区域 |

ASIL 等级直接影响算法选择。ASIL-B 通常要求 90% 以上的单点故障指标，March C- 或 March B 即可满足。ASIL-D 可能要求更高的故障检测覆盖率，需要组合多种算法或使用 GALPAT 测试关键区域。

## 前台与后台执行模式

### 前台测试（Foreground）

前台测试通过 `RamTst_RunFullTest` 或 `RamTst_RunPartialTest` API 同步调用，执行期间不可被中断。它的优势是测试过程原子性完整——没有中断带来的时间窗口，不会出现测试期间 RAM 被其他任务修改的情况。前台测试适合两个时机：ECU 启动阶段（上电自检）和工厂下线检测。

### 后台测试（Background）

后台测试通过周期性调用 `RamTst_MainFunction` 异步执行。每次调用只测试配置数量的 RAM 单元，多次调用后完成整个区域的测试。这种分片执行的机制让 RAM 测试可以融入正常的任务调度，不影响高优先级任务的实时性。

实际项目中常见的做法是：启动时用前台测试快速验证全部 RAM，运行时用后台测试的 Transparent 算法周期性验证关键变量区域。两层配合，既保证了启动时的全面覆盖，又不影响运行时的实时性。

## 实践建议

- 启动时用前台 March C- 全量扫描，运行时用后台 Transparent 算法做增量检查
- 后台测试的单次测试单元数直接影响 MainFunction 的执行时间。设太小完成一轮测试的周期太长，设太大可能阻塞同优先级任务。建议测量单次调用的 WCET，留出 30% 余量
- 不是所有 RAM 都需要测试。根据安全分析确定哪些区域存放 ASIL 相关的变量，只在那些区域配置 RamTst
