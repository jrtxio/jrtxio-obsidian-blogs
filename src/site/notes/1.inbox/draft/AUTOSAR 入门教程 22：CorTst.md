> 理解处理器核心测试的各个部分测试项（ALU、寄存器、中断控制器等），掌握前台与后台执行模式的选择，以及如何按 ASIL 等级配置测试范围。

RAM 测试验证存储器硬件是否可靠，CorTst（Core Test）则验证处理器核心本身是否正常工作。如果 ALU 的加法器出了故障，或者某个通用寄存器卡死在 0，程序的计算结果就是错的——而这种错误不会被编译器或操作系统发现。在 [[3.wiki/AUTOSAR\|AUTOSAR]] MCAL 层中，CorTst 模块专门负责检测这类处理器硬件故障，是功能安全认证中的重要环节。

## 部分测试项详解

CorTst 由一组部分测试（Partial Tests）组成，每个部分测试验证处理器的一个特定功能单元。部分测试具有原子性——要么完整执行，要么不执行，不允许中途被打断后只完成一半。

### ALU 测试

ALU（算术逻辑单元）是处理器执行运算的核心部件。ALU 测试的目标是验证运算逻辑的正确性，策略是针对每类运算选择一组能最大化覆盖硬件逻辑路径的测试向量。

以加法器测试为例：向 ALU 输入 `0x55555555 + 0xAAAAAAAA`，期望结果为 `0xFFFFFFFF`（全 1，无进位）。再输入 `0xFFFFFFFF + 0x00000001`，期望结果为 `0x00000000`（全零，进位溢出）。这两组测试向量分别覆盖了加法器中"每个 bit 位都有 0 和 1 的翻转"以及"进位链完整传播"两条关键路径。

ALU 测试还涵盖标志位验证：零标志（Zero Flag）、负数标志（Negative Flag）、进位标志（Carry Flag）、溢出标志（Overflow Flag）。每条运算指令执行后，检查标志位是否符合预期。如果加法器结果正确但零标志位卡死，条件分支指令就会误判。

### 寄存器测试

寄存器测试验证通用寄存器和特殊寄存器的读写正确性。常用的测试模式包括 Walk Pattern（在一个全 0 的寄存器中逐位写入 1 并读回验证）和 Checkerboard（交替写入 `0x55555555` 和 `0xAAAAAAAA`，验证相邻 bit 之间是否存在短路或耦合故障）。

通用寄存器测试相对直接——保存当前值，写入测试模式，读回验证，恢复原始值。特殊寄存器更复杂：栈指针不能随意修改，程序计数器无法直接写入测试模式，状态寄存器的某些位是只读的。这些限制决定了特殊寄存器的测试需要使用间接方法——通过特定指令序列触发状态变化，再读取对应状态位验证。

### 中断控制器与 MPU 测试

**中断控制器测试**验证中断请求、优先级仲裁和中断向量寻址的正确性。测试方法是在安全状态下触发软件中断，验证中断服务程序是否被正确调用、返回后上下文是否完整恢复。**MPU（Memory Protection Unit）测试**验证内存保护规则的正确执行，尝试访问被 MPU 标记为不可访问的内存区域，检查是否正确触发了 MemManage 异常。

## 执行模式与 ASIL 对应

### 前台与后台

前台测试通过 `CorTst_Start()` API 触发，同步执行全部已使能的部分测试。执行期间不可被中断，通常在 ECU 启动阶段或安全关键操作前调用。前台测试的优势是测试环境干净——没有其他任务干扰，测试结果可信度高。

后台测试通过周期性调用 `CorTst_MainFunction` 执行，每次调用完成一个或多个部分测试。由于可被高优先级任务中断，执行时间跨度更大，但不会阻塞正常功能。后台测试完成后，通过配置的回调函数通知调用方测试结果。

### ASIL 等级与测试范围

功能安全等级直接决定 CorTst 的配置范围：ASIL-B 通常要求 ALU 测试和寄存器测试，其他测试项可以按需使能；ASIL-D 要求更全面的测试覆盖，包括 ALU、寄存器、中断控制器、内存接口等，部分标准还要求在运行时周期性执行核心测试。

由于芯片制造商最了解自家处理器核心的架构特性，CorTst 模块通常由芯片厂商提供实现。开发者通过配置界面使能/禁用各测试项，调整测试顺序和超时参数。芯片厂商的安全手册（Safety Manual）会明确列出每个 ASIL 等级需要执行哪些测试项。

## 实践建议

- ECU 上电后、应用逻辑启动前，调用 `CorTst_Start()` 执行完整的处理器核心测试。如果处理器核心本身有问题，后续所有运算都不可信
- 按安全手册配置测试项，不要自行裁剪测试范围。严格按芯片厂商的安全手册配置
- 关注测试执行时间：ALU 和寄存器测试的执行时间可能较长。在后台测试模式下，确保 `CorTst_MainFunction` 的 WCET 不会影响高优先级任务的截止时间
